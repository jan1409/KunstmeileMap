# Excel-Export für Stände und Kategorien — Design Spec

**Date:** 2026-05-27
**Status:** Approved, ready for implementation plan
**Author:** Brainstorm session with Jan
**Target branch:** new feature branch `feat/excel-export` from `main` (current main tip: `16dca51`, the squash-merged map pivot)

## Context and rationale

The admin CMS already has a bulk-import wizard for stands (`/admin/events/<slug>/tents/import`) that accepts `.csv` and `.xlsx`. Admins can populate ~100 stands via the wizard and then place them on the map. What's missing is the inverse: there's no way to extract the current state of stands or categories back out to a spreadsheet for backup, offline editing, sharing with the organizer team, or simply as a snapshot.

This spec adds two small export buttons — one for stands, one for categories — each producing a date-stamped `.xlsx` download with columns mirroring the existing import format (round-trip-friendly).

This is a Phase-1-post-launch convenience feature. The user requested it as the first of two follow-up features after the map pivot landed (the second, role management, gets a separate spec).

## Out of scope

- **Kategorien-Import** (the inverse of the new category export). The existing import wizard handles tents only. Categories are managed via `CategoryListPage`. If category-import becomes useful later, it's a separate spec.
- **Photo export.** Tent photos live in Supabase Storage and don't fit the spreadsheet model. Backup of photos is a separate concern.
- **Whole-event JSON backup** or DB dump. Out of scope here.
- **Excel formulas, multi-sheet workbooks, formatting.** The export is plain data: a header row and data rows in the same column order the import wizard expects. No styling, no formulas, no merged cells.
- **CSV export.** XLSX is the only export format. CSV import is preserved on the input side; export doesn't mirror it.

## Locked decisions (from brainstorming)

| # | Decision |
|---|----------|
| EX-001 | Two separate `.xlsx` files (not one workbook with two sheets, not a single combined download). One button per list page. |
| EX-002 | Stands export columns match the import wizard's `RawRow` expectations exactly, so an exported file is directly re-importable without manual reformatting: `name, display_number, slug, category_slugs, description_de, description_en, address, website_url, instagram_url, facebook_url, email_public, lat, lng`. |
| EX-003 | Categories export columns: `slug, name_de, name_en, icon, display_order`. Matches the `categories` table schema (minus internal-only fields like `id`, `event_id`, `created_at`). |
| EX-004 | Both exports include real values for `display_number` and `slug`, even when those values were originally auto-generated by the DB trigger / import wizard. Round-trip works without surprises. |
| EX-005 | Empty / null cells stay empty in the .xlsx (not `null` strings, not `--`, not `N/A`). |
| EX-006 | Filename convention: `kunstmeile-<event-slug>-tents-YYYY-MM-DD.xlsx` and `kunstmeile-<event-slug>-categories-YYYY-MM-DD.xlsx`. Date stamp helps admins keep multiple snapshots distinguishable. |
| EX-007 | Button placement: in the page header next to the existing primary actions. `TentListPage` gets "↓ Excel-Export" next to "+ Neuer Stand" / "CSV-Import". `CategoryListPage` gets "↓ Excel-Export" next to "+ Neue Kategorie". |
| EX-008 | No new dependencies. SheetJS (`xlsx`) is already in the project (added during the map pivot for `.xlsx` import). |
| EX-009 | No schema changes. Pure read-side feature. |
| EX-010 | i18n strings are added under `admin.tent_list.export_xlsx` and `admin.category.export_xlsx`. Both DE and EN. |

## Data model changes

**None.** The export reads from existing `tents` and `categories` tables via existing hooks (`useTents`, `useCategories`).

## Behavior

### Tents export

When the admin clicks "↓ Excel-Export" on the tent-list page:

1. The current event's tents (already loaded via `useTents(event.id)`, which returns `TentWithCategories[]` with each tent's joined categories array) are passed to `exportTentsToBlob`. No separate categories list is needed — each tent already carries its category slugs.
2. The helper builds an array of row objects — one per tent — with the 13 columns from EX-002 in order.
3. `category_slugs` is joined from each tent's `tent_categories → categories.slug` association as a comma-separated string. Tents with zero categories produce an empty string in that column.
4. `lat` and `lng` are written as numbers when set, empty cells when null.
5. SheetJS produces an .xlsx workbook with one sheet (named `Tents`).
6. The browser triggers a download with filename `kunstmeile-<event.slug>-tents-<YYYY-MM-DD>.xlsx`.

### Categories export

When the admin clicks "↓ Excel-Export" on the category-list page:

1. The current event's categories (via `useCategories(event.id)`) are passed to `exportCategoriesToBlob`.
2. The helper builds row objects with the 5 columns from EX-003.
3. Rows are ordered by `display_order` ascending (matches the in-app display order).
4. SheetJS produces an .xlsx workbook with one sheet (named `Categories`).
5. Filename: `kunstmeile-<event.slug>-categories-<YYYY-MM-DD>.xlsx`.

### Empty event

If an event has zero tents (or zero categories), the export still produces a valid .xlsx with only the header row. Admin sees a working download instead of an error.

### File-trigger pattern

```ts
function downloadBlob(blob: Blob, filename: string) {
  const url = URL.createObjectURL(blob);
  const a = document.createElement('a');
  a.href = url;
  a.download = filename;
  document.body.appendChild(a);
  a.click();
  document.body.removeChild(a);
  URL.revokeObjectURL(url);
}
```

This pattern is browser-portable and doesn't require any extra library. Lives as a small helper either in `src/lib/download.ts` (new tiny file) or inlined in each page — small enough to inline if only used twice. Plan favors the inlined-in-page approach unless a third caller appears.

## Round-trip guarantee

The whole-feature acceptance test is: **export a populated event's tents, change one row in Excel, re-import via the existing wizard, the modified row arrives in the DB with the change applied.**

To make this work in practice:

- Exported `display_number` is the actual stored integer, not the original "blank for auto-assign" placeholder.
- Exported `slug` is the actual stored slug (the import wizard auto-generates from `name` when blank — round-trip preserves stable slugs by always including them).
- `lat` / `lng` round-trip as numbers; the wizard accepts numeric cells via the existing SheetJS coercion.
- `category_slugs` round-trips through the comma-separated form — admin can edit the string in Excel, re-import maps slugs back to category IDs (the wizard logic for unknown-slug warnings is unchanged).
- Empty optional fields round-trip as empty strings (the wizard already tolerates empty for the optional URL/email/coord columns).

Round-trip is not enforced by tests beyond manual smoke (the export tests verify produced shape; the import tests verify consumption). A combined integration test is YAGNI for this feature.

## Test strategy

Per the project's `feedback_workflow.md`:

- **TDD for the pure helpers** in `src/lib/excel.ts`. Each new function gets failing tests first, then minimal implementation.
- **No new component tests** for the page-level button wiring — the button just calls the helper and triggers a download. The existing TentListPage and CategoryListPage tests already cover the surrounding UI; we add nothing new there unless lint or type-check forces an adjustment.
- **Manual smoke** checklist update in `tests/manual/map-pivot-smoke.md` (or a new dedicated smoke file — implementer's choice). Bullets:
  - Click "↓ Excel-Export" on the tent list → file downloads with the right name.
  - Open the file in Excel → header row + one row per tent, columns match the import format.
  - Open with zero tents → file downloads, header-only, no error.
  - Same for categories.
  - Round-trip: export → tweak one row → re-import → verify the tweak landed.

## Risks and mitigations

| Risk | Mitigation |
|---|---|
| Exported file too large for Excel to handle on slow devices | For ~100 stands the file is < 50KB. Non-issue at current scale. |
| Filename collisions when admin exports twice on the same day | Browser appends `(1)`, `(2)` etc. Acceptable. If a sub-second timestamp is wanted later, swap `YYYY-MM-DD` for `YYYY-MM-DD-HHmm`. |
| User opens .xlsx in a tool that mangles category_slugs string (e.g., turns `painting,sculpture` into two columns) | This is a user-side concern. We can't prevent it. Acceptable. |
| Re-imported file with edited slug collides with an existing slug | The import wizard's existing per-row validation flags `slug duplicated`. Handled. |

## Acceptance criteria

Feature ships when:

1. `pnpm test:run`, `pnpm type-check`, `pnpm lint`, `pnpm build` all clean.
2. The new export tests in `tests/unit/lib/excel.test.ts` cover at minimum: column order, comma-separated `category_slugs`, null-cell-empty handling for both helpers.
3. Manual smoke passes: download both files, verify shape, round-trip a tent.
4. PR squash-merges to main without conflicts.
